Git与代码托管平台入门教程

欢迎来到Git教程！本教程旨在帮助零基础的同学快速理解什么是Git，为什么我们使用它，以及它如何与代码托管平台（如 GitHub）协同工作。

1. 什么是版本控制？

想象一下你正在写一篇论文。你写了第一版，保存为 paper_v1.doc。然后你修改了它，保存为 paper_v2.doc。过了几天，你又改了，保存为 paper_final.doc... 然后是 paper_final_REALLY_final.doc。

这个过程非常混乱且容易出错。版本控制系统（Version Control System, VCS）就是用来解决这个问题的。它会帮你记录你对文件所做的每一次修改，这样你就可以：

  • 查看历史记录
  • 回溯到任何一个旧版本
  • 比较不同版本之间的差异
  • 与他人协同工作而不会互相覆盖

2. 什么是Git？

Git是目前世界上最先进、最流行的分布式版本控制系统。简单来说，它是一个能帮你管理代码（或任何文件）历史版本的强大工具。

与旧的集中式系统不同，Git是“分布式”的。这意味着每个开发者（包括你！）的电脑上都有一个完整的代码仓库，包含所有的历史记录。这使得离线工作和分支管理变得极其高效。

3. Git 的核心概念与工作流程

理解Git，你首先需要知道三个“区域”：

  1. 工作区 (Working Directory): 你在电脑上实际看到和编辑的文件。
  2. 暂存区 (Staging Area): 一个“准备提交”的区域。你把确认无误的修改“添加”到这里。
  3. 本地仓库 (Local Repository): 你的个人数据库，存储了所有的提交历史（快照）。

基本工作流程:

你日常使用Git的流程通常是这样的：

  1. 在工作区修改文件。
  2. 使用 git add 命令，将修改添加到暂存区。
  3. 使用 git commit 命令，将暂存区的内容“提交”到本地仓库，形成一个永久的快照。

这就是“实验 2: 第一次提交”中练习的核心流程！

5. 核心知识点（对应9大实验）

本教程的知识点与实验紧密相连。你可以在这里预习，或在实验后复习。

实验 1: 创世纪 (init & config)

这是你 Git 旅程的起点。git init 会在你的项目文件夹中创建一个隐藏的 .git 目录。这个目录就是你的“本地仓库”，它包含了 Git 用来跟踪版本所需的所有信息。

git config 则是设置你的身份。每次你提交代码时，Git 都会记录下是谁（user.name）和联系方式（user.email）做的这次提交。设置成 --global 意味着它会应用到你电脑上的所有项目中。

实验 2 & 3: 提交与检查 (add, commit, status, log)

这是 Git 最核心的工作流。git status 是你的仪表盘，它告诉你当前哪些文件被修改了（工作区），哪些文件准备好了被提交（暂存区）。

git add . 将你所有的修改从工作区“快照”到暂存区。git commit -m "message" 则将暂存区里的所有内容打包，创建一个永久的“提交记录”到本地仓库。这个 "message" 至关重要，它简明扼要地描述了你这次提交做了什么。

git log 允许你查看所有的提交历史，就像查看你项目的历史账本一样。

实验 4: 时间旅行 (checkout & reset)

Git 最强大的功能之一就是“后悔药”。

  • git checkout <commit-hash>: 让你能“穿越”回任意一次提交，查看当时项目的状态（只读）。
  • git reset <commit-hash>: (需谨慎！) 彻底地回溯到某个版本，抛弃这之后的所有修改。这是真正的“重置”。

实验 5: 平行宇宙 (branch & merge)

分支（Branch）是 Git 的“杀手级”功能。想象一下，你要开发一个新功能，但又不想影响主线代码（通常是 master 或 main 分支）。

你可以使用 git branch feature-x 创建一个名为 "feature-x" 的新分支。这个分支是当前代码的一个完整副本。你可以在这个“平行宇宙”里随心所欲地修改、提交，而主分支不受任何影响。

当你完成了新功能，就可以使用 git merge feature-x 将这个分支的修改“合并”回主分支。

实验 6: 连接远程 (remote, push, pull)

这就是代码托管平台（如 GitHub）发挥作用的地方。你的本地仓库只存在于你的电脑上。远程仓库（Remote）是托管在云端的“中央服务器”。

  • git remote add origin <url>: 告诉你的本地仓库，远程仓库的地址在哪里（通常命名为 origin）。
  • git push: 将你本地的提交“推送”到远程仓库，与他人分享。
  • git pull: 从远程仓库“拉取”别人推送的最新更改，与你本地代码合并。

实验 7: 团队协作 (fetch & rebase)

git pull 实际上是两个命令的组合：git fetch 和 git merge。

  • git fetch: 只是“下载”远程的最新更改，但*不*合并。它让你先看看别人做了什么。
  • git rebase: 这是 merge 的一种替代方案。它会把你本地的提交“拎起来”，放到远程最新提交的“顶部”，使提交历史看起来像一条直线，非常整洁。

实验 8: 解决冲突 (Conflict Resolution)

当两个（或更多）人同时修改了*同一个文件*的*同一行*代码时，Git 无法自动合并。这时就会发生“合并冲突”(Merge Conflict)。

Git 会停止合并，并在冲突文件中标记出问题区域。你需要手动打开这个文件，决定保留哪部分代码（或者都保留，或者都删除），然后再次 git add 和 git commit 来告诉 Git：“我已经解决了这个冲突”。

实验 9: 开源协作 (Fork & Pull Request)

当你想要为一个你“不拥有”的开源项目（例如 GitHub 上的项目）做贡献时，你就需要用到 Fork 和 Pull Request (PR) 工作流。

什么是 Fork？

Fork（分叉） 是一个 GitHub（或类似平台）上的操作。它会在你自己的账户下完整地“复制”一份别人的项目。这份副本完全归你所有，你可以随意修改它，而不会影响原项目。

什么是 Pull Request (PR)？

Pull Request (PR)，即“拉取请求”，是协作的核心。当你 Fork 了项目并做出了有价值的修改后，你可以向“原项目”发起一个 PR。

这个 PR 就像在说：“你好，原项目作者！我在我的副本里做了一些改进（比如修复了一个 Bug），你是否愿意‘拉取’(pull) 我的这些更改，并把它们合并到你的主项目中？”

原项目的维护者会收到通知，他们可以审查你的代码（Code Review），提出修改意见，如果一切OK，他们就会点击“合并”，你的贡献就正式成为原项目的一部分了！

标准流程:

  1. 在 GitHub 上 Fork 目标项目。
  2. Clone 你自己 Fork 后的项目到本地：git clone <your-fork-url>
  3. 创建新分支：git checkout -b my-fix
  4. 修改代码、git add、git commit。
  5. 推送到你的 Fork 仓库：git push origin my-fix
  6. 最后，访问 GitHub，平台会自动检测到你的新分支，并提示你 "Create a Pull Request"。

6. Git 最佳实践指南

掌握了命令之后，如何“优雅”地使用 Git 同样重要。良好的 Git 习惯是高效团队协作的基石。

Commit Small, Commit Often (少量多次提交)

不要等到开发完一个巨大功能再提交。应该将功能拆分为小的、逻辑相关的步骤，每完成一步就进行一次提交。这样做的好处是：

  • 更容易写出清晰的提交信息。
  • 如果出现 Bug，能更快地定位到问题所在（git bisect）。
  • 更容易进行代码审查（Code Review）。

Write Good Commit Messages (编写规范的提交信息)

一个糟糕的提交信息（如 "fix bug" 或 "update"）是无用的。一个好的提交信息应该遵循一定格式，例如：

feat: Add user login functionality

- Implement email/password validation on the server.
- Create login form component on the client.
- Add JWT authentication middleware.
  • 标题 (Subject): 简明扼要，使用祈使句（如 "Add" 而不是 "Added"）。通常以类型开头（如 feat:, fix:, docs:）。
  • 正文 (Body): (可选) 详细解释“为什么”要这样改，而不是“怎么”改。

Pull Often, Especially Before Pushing (经常拉取，推送前必拉取)

在开始新工作或推送你的更改之前，养成 git pull（或 git fetch + git rebase）的习惯。这能确保你的本地分支基于最新的远程代码，最大限度地减少合并冲突。

Never Push Broken Code (绝不推送未完成或损坏的代码)

推送到共享分支（如 master 或 development）的代码应该是可以运行的。在推送前，至少要在本地完整测试你的更改。破坏主分支会阻碍团队所有人的工作。

Use Branches for Everything (为所有功能使用分支)

永远不要直接在 master 分支上开发！master 分支应该始终保持稳定且可部署。任何新功能、Bug 修复，哪怕只是一行代码的修改，都应该在一个新的特性分支（Feature Branch）上进行，完成后再通过 Pull Request 合并。

7. 常见问题 (FAQ)

Q1: `git add .` 和 `git add -A` 有什么区别?

在现代 Git 版本中，它们基本相同，都会暂存所有“新文件”、“修改的文件”和“删除的文件”。在旧版本中，git add . 可能不包括被删除的文件。

Q2: 我刚刚的 commit 提交错了，如何撤销？

如果你只是想修改最后一次提交的信息（还没 push），使用：

$ git commit --amend -m "New, correct commit message"

如果你想彻底撤销最后一次提交（还没 push），让更改退回暂存区：

$ git reset --soft HEAD~1

如果你想彻底撤销最后一次提交，并且丢弃所有更改（危险操作！）：

$ git reset --hard HEAD~1

Q3: 什么是 `.gitignore` 文件？

.gitignore 是一个纯文本文件，你可以在里面列出不希望 Git 跟踪的文件或目录。Git 会自动忽略这些文件，它们不会出现在 git status 中，也不会被 git add . 添加。

常见忽略内容：

  • 依赖包（如 node_modules/）
  • 编译产生的文件（如 .class, .exe）
  • 操作系统文件（如 .DS_Store, Thumbs.db）
  • 敏感信息（如 .env, config.log）

Q4: 我应该用 `merge` 还是 `rebase`?

这是一个“圣战”话题，但通常的建议是：

  • Merge: 默认选项。它会保留分支的完整历史，产生一个“合并提交” (Merge Commit)。历史更“真实”，但可能更杂乱。
  • Rebase: 高级选项。它会重写你的提交历史，使其呈一条直线。历史非常“干净”，易于阅读。

黄金法则： 永远不要对一个已经推送到远程、并且团队其他人可能正在使用的公共分支（如 master）进行 rebase。
